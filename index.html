Surgical Partners — Fund Administration Platform

Feb 2025 – Present

Full Stack Developer

Banking Integration Partner — Commonwealth Bank of Australia

Full-stack engineering on a fund-administration platform for surgical and healthcare providers, spanning two generations of frontend architecture and a .NET/SQL Server backend engineered for high-volume, secure financial transaction processing.

The Brief

The platform runs core fund-administration operations for healthcare providers — billing, disbursements, bank statement reconciliation, guarantees, fee runs, and time sheets — where correctness of financial data, strict access control, and reliability under transaction volume matter more than raw feature velocity. The bank statement reconciliation module works directly against data from Commonwealth Bank of Australia, the country's largest bank, which raised the bar on data accuracy and compliance beyond what the rest of the platform required. The problem was delivering these modules on a UI layer that could evolve architecturally over the project's lifetime without sacrificing design consistency, while keeping the backend fast and auditable enough for high-volume financial workloads.

The Build

I worked across the full stack, contributing to two successive frontend implementations: an initial version in React with Redux for state management and PrimeReact for UI components, and a subsequent rebuild in Angular 20 using NgRx (Store, Effects, Entity) alongside Angular Signals, PrimeNG, and Tailwind CSS. Across both versions I built reusable UI component libraries to keep design and interaction patterns consistent as modules were added, and used Chart.js for dashboard and reporting views that visualize complex financial data.

The backend runs on C#/.NET Core with ASP.NET Core Web APIs and Entity Framework Core against SQL Server, with hand-optimized stored procedures, queries, and indexes to keep high-volume transaction processing performant. Authentication is JWT-based with role/claim-based access control, integrated with Azure AD via MSAL, with secrets managed through Azure Key Vault. Workflows that shouldn't run synchronously go through Azure Service Bus messaging paired with .NET background worker services, with retry and error-handling logic to keep processing reliable. Observability runs through Azure Application Insights and Sentry, with xUnit covering backend unit tests. The reconciliation module ingests statement and transaction data sourced from Commonwealth Bank of Australia, matching it against internal ledger records as part of the platform's reconciliation workflow.

The Hard Part

Making the platform's asynchronous, message-based financial workflows resilient rather than just functional. Background operations driven by Azure Service Bus needed to survive transient failures — network blips, downstream timeouts, dependent service hiccups — without silently dropping or duplicating financial data. I built the background worker services with explicit retry and error-handling logic around each processing stage rather than relying on default queue behavior, paired with Application Insights and Sentry instrumentation so a production failure could be traced back to the specific message and workflow step instead of surfacing only as a downstream data discrepancy.

The Outcome

Delivered and maintained production functionality for billing, disbursements, bank statement reconciliation, guarantees, fee runs, and time sheets on a live platform. Reusable component libraries kept UI/UX consistent across a full frontend framework migration from React/Redux to Angular/NgRx, and the combined observability setup improved the team's ability to detect and resolve production issues quickly on a system handling sensitive financial data.

Themis — Regulatory Compliance Platform

May 2023 – Dec 2025

Full Stack Developer

A regulatory-compliance platform built end-to-end over two years, spanning a reactive Angular front end, a layered .NET backend, and an automated document-generation pipeline that turns calculation results into audit-ready compliance paperwork.

The Brief

The client needed a platform to support compliance-driven workflows for an organization operating under regulatory oversight — one that could run calculation logic consistently, produce formal compliance documents from that logic without manual rework, and do so securely across DEV, UAT, and production for both interactive users and background service processes.

The Build

The frontend is Angular 19 with PrimeNG and RxJS for reactive state and async data flows. The backend follows a layered architecture on ASP.NET Core 8, exposing Web APIs for interactive traffic and a separate .NET Worker Service for background processing such as document generation, keeping heavy work off the request path of the main API. Data access runs through Entity Framework Core 8 against SQL Server. Identity is handled via Azure AD B2C with MSAL on the client and JWT validation via Microsoft.Identity.Web on the API, while trusted internal calls use a separate API-key scheme, with Azure Key Vault centralizing secrets. The platform's reach was extended through Azure Communication Services, Azure Maps, and Microsoft Graph API. Observability combines Azure Monitor with OpenTelemetry and Sentry, with MSTest covering backend unit tests.

The Hard Part

Automated generation of regulatory compliance documents: calculation engines produced the figures, and those results had to be rendered into consistent, audit-ready PDFs without manual assembly. I built this with QuestPDF, keeping calculation logic in the domain/application layers so it stayed independent of and reusable outside the rendering step. Because document generation could be resource-intensive and didn't need to block a user's request, I routed it through the .NET Worker Service rather than the synchronous Web API — which also meant designing two separate auth paths suited to their trust levels (end-user JWTs vs. service-to-service API keys) without duplicating secret management.

The Outcome

Over the nearly two-year engagement, the platform matured into a stable production system spanning DEV, UAT, and production, with automated PDF generation replacing manual compilation of compliance documents. Beyond feature delivery, I worked directly with stakeholders on requirements and estimation, triaged and resolved defects across all environments, and mentored other developers on the team.

Network Courier — Logistics ERP Platform

May 2022 – Jun 2025

Frontend Developer

Frontend development on a multi-domain courier and logistics ERP platform, building modular Vue.js features across sales, fleet operations, billing, CRM, HR, and warehouse management for a system spanning over 100 store modules.

The Brief

The platform needed to support a wide range of operational domains — sales, fleet operations, billing, CRM, HR, and warehouse management — for a courier and logistics business, all within a single, modular ERP built as a Vue.js single-page application. With over 100 Vuex store modules across the platform, the challenge was keeping API integration, authentication, and UI patterns consistent across a large, multi-domain codebase rather than letting each area reinvent its own conventions.

The Build

I worked within a modular, namespaced Vuex 4 store architecture (Sales, Operations, Fleet Management, Billing, CRM, HRM, Warehouse) using Vue 3 and Vue Router 4, developing and maintaining feature modules across these domains. I built a centralized Axios-based API service layer with request/response interceptors handling bearer-token authentication and global loading-state, standardizing REST integration across 100+ Vuex store modules, and implemented token-based route guards via Vue Router integrated with role/module-based permission screens. I also built and maintained a shared component library (data tables, dropdowns, dialogs, steppers, file uploads) on PrimeVue, PrimeFlex, and Bootstrap 5 to keep UI/UX consistent across the platform's many modules.

The Hard Part

Real-time fleet and rider tracking needed to feel responsive and accurate for live operations monitoring, not just a static map view. I implemented this using the Google Maps JavaScript API with marker clustering, live GPS coordinate updates, and optimized route retrieval, so dispatchers could track delivery riders across the platform in real time without the map degrading under load.

The Outcome

The shared component library and centralized API layer meant new domains and features could be added without re-solving the same authentication, loading-state, and UI-consistency problems. I also contributed to multi-step quotation and contract creation workflows (Express, International, Rapido service lines) using Vuelidate form validation, and built scheduling and reporting dashboards using FullCalendar and Chart.js for operational visibility.
